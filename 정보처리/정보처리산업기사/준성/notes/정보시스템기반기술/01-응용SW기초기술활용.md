# 01. 응용SW기초기술활용

> 출처: `_공통/출제기준/정보시스템기반기술.md`의 주요항목 순번을 따른다.

## 운영체제 기초 활용

### 핵심 개념
- 정의: 프로세스 스케줄링 — 실행 대기 중인 여러 프로세스 가운데 어떤 것에 CPU를 먼저 배정할지 순서를 정하는 작업. (프로세스 = 현재 실행 중인 프로그램)
- 정의: 디스크 스케줄링 — 디스크 헤드가 트랙(실린더)을 이동하면서 대기 큐의 입출력 요청을 어떤 순서로 처리할지 결정하는 알고리즘. (프로세스 스케줄링이 "CPU 배정 순서"라면, 디스크 스케줄링은 "헤드 이동 순서")
- 왜 중요한가: 스케줄링 알고리즘을 선점형/비선점형으로 분류해 짝짓는 문제, 디스크 대기 큐에서 처리 순서·다음 트랙을 계산하는 문제가 기출 단골.

### 상세 정리

**선점형 vs 비선점형**

- **비선점형(Non-preemptive)**: 한 프로세스가 CPU를 차지하면 그 작업이 끝날 때까지 다른 프로세스가 CPU를 강제로 뺏을 수 없는 방식. 동기/비동기와는 다른 개념 — "진행 중인 작업을 다 끝내야만 다음 작업으로 넘어간다"는 뜻.
- **선점형(Preemptive)**: 실행 도중이라도 운영체제가 CPU를 강제로 회수해 다른 프로세스에 줄 수 있는 방식.

| 방식 | 선점 여부 | 동작 규칙 |
|------|----------|-----------|
| FIFO (FCFS) | 비선점 | 먼저 도착한(First In) 프로세스부터 처리 |
| SJF (Shortest Job First) | 비선점 | CPU가 비었을 때 대기열에서 실행시간이 가장 짧은 작업을 골라 끝까지 실행 |
| SRT (Shortest Remaining Time) | 선점 | 새 프로세스 도착 시 "남은 시간"을 비교해 더 짧은 쪽에 CPU를 뺏어줌 |
| Round-Robin | 선점 | 모든 프로세스가 Time Quantum(시간 할당량)만큼만 번갈아 사용, 시간 초과 시 강제 교체 |

**SJF가 비선점형인 이유 (헷갈림 주의)**

- "가장 짧은 것부터"라는 규칙은 **CPU가 비어 있을 때(현재 작업이 완전히 끝났을 때)만** 적용된다.
- 실행 도중에 더 짧은 작업이 도착해도 절대 뺏기지 않고 하던 작업을 끝낸 뒤, 그때 대기열에서 최단 작업을 고른다. → 선택 "타이밍"이 시작 시점뿐이므로 비선점형.

**SRT의 '남은 시간' 의미**

- 여러 작업을 동시에 시작해서 남은 시간을 보는 게 아니라, **현재 실행 중인 프로세스가 끝날 때까지 남은 시간**을 뜻한다.
- 새 프로세스가 도착할 때마다 「실행 중 작업의 잔여 시간」 vs 「새 작업의 소요 시간」을 비교해 더 짧은 쪽으로 CPU를 즉시 넘긴다(선점).
- 예: A(10분짜리)를 2분 진행해 잔여 8분인 시점에 B(3분짜리) 도착 → SJF는 A를 마저 끝내지만, SRT는 A를 중단하고 B를 먼저 실행.

**디스크 스케줄링 기법 총정리**

| 기법 | 처리 방식 | 특징 |
|------|-----------|------|
| FCFS (FIFO) | 대기 큐에 들어온 순서대로 처리 | 공평하지만 헤드가 앞뒤로 크게 움직여 효율 낮음 |
| SSTF (Shortest Seek Time First) | 현재 헤드에서 탐색 거리(Seek Time)가 가장 짧은 = 가장 가까운 트랙부터 처리 | 처리량 높음. 먼 요청이 계속 밀리는 **기아 상태(Starvation)** 발생 가능 |
| SCAN (엘리베이터) | 한쪽 끝까지 이동하며 진행 방향의 모든 요청 처리, 끝에 도달하면 방향 전환 | 엘리베이터처럼 동작. SSTF의 기아 상태 완화 |
| C-SCAN (Circular SCAN) | 항상 한 방향으로만 처리. 끝에 도달하면 반대쪽 처음(0번)으로 빠르게 복귀(복귀 중 처리 안 함) 후 같은 방향 반복 | 양끝 요청이 오래 기다리는 SCAN 단점 보완 → 응답 시간 편차 감소 |
| LOOK | SCAN과 같되, 진행 방향에 더 이상 요청이 없으면 끝까지 가지 않고 **즉시 방향 전환** | SCAN의 불필요한 끝점 이동 제거 |
| C-LOOK | C-SCAN과 같되, 마지막 요청 처리 후 끝(199·0번)까지 가지 않고 **가장 낮은 요청 위치로 바로 점프** | C-SCAN의 불필요한 끝점 이동 제거 |
| N-Step SCAN | 이동 시작 시점에 큐에 있던 요청만 한 방향으로 처리, 도중 도착한 요청은 다음(반대 방향) 이동 때 처리 | 특정 트랙 집중 요청 시의 무한 대기를 확실히 방지 |
| 에센바흐(Eschenbach) | 헤드 이동 시간(Seek Time) + **회전 지연 시간(Rotational Delay)**까지 고려해 큐 재배열 | 탐색·회전 최적화를 동시에 노림 |

**공통 예시로 보는 처리 순서** (헤드 53, 대기 큐 98·183·37·122·14·124·65·67, 트랙 0~199, 증가 방향 이동 중)

| 기법 | 처리 순서 |
|------|-----------|
| FCFS | 53 → 98 → 183 → 37 → 122 → 14 → 124 → 65 → 67 (큐 순서 그대로) |
| SSTF | 53 → 65 → 67 → 37 → 14 → 98 → 122 → 124 → 183 (매번 최근접: 67에서 37(30) < 98(31)이라 37로) |
| SCAN | 53 → 65 → 67 → 98 → 122 → 124 → 183 → **199(끝 도달, 전환)** → 37 → 14 |
| C-SCAN | 53 → 65 → 67 → 98 → 122 → 124 → 183 → **199** → **0(복귀, 처리 없음)** → 14 → 37 |
| LOOK | 53 → 65 → 67 → 98 → 122 → 124 → **183(마지막 요청, 전환)** → 37 → 14 |
| C-LOOK | 53 → 65 → 67 → 98 → 122 → 124 → **183** → **14(최저 요청으로 점프)** → 37 |

**기출 적용 — 헤드 진행 방향 읽어내기**

- 문제: 헤드가 70 트랙을 처리하고 60 트랙으로 이동해 옴. SCAN에서 대기 큐 20·50·95·100 중 가장 먼저 처리되는 트랙은?
- 풀이: 70 → 60 이동 = **트랙 번호 감소 방향으로 진행 중**. SCAN은 끝(0번)에 닿기 전까지 진행 방향을 유지하므로, 60 아래의 요청 중 가장 가까운 **50**이 먼저 처리된다. (이후 20 → 0 → 방향 전환 → 95 → 100)
- 내 오답 원인: 방향을 증가로 착각하면 95를 고르게 된다. SCAN 계산 문제는 **직전 이동(70→60)에서 현재 방향부터 판정**하는 게 첫 단계.

### 암기 포인트
- ⭐ 비선점 = **SJF, FIFO** / 선점 = **SRT, Round-Robin**. 이름에 R이 들어간 SRT·RR은 "뺏는(선점)" 쪽으로 기억.
- ⭐ SJF와 SRT의 차이는 "언제 고르느냐": 끝나고 고르면 SJF(비선점), 도착할 때마다 다시 재면 SRT(선점).
- ⭐ 디스크 스케줄링 이름 해석: **SCAN=엘리베이터(끝까지 갔다 반대로)**, **LOOK=요청 없으면 바로 회차(끝까지 안 봄)**, **C-=Circular(한 방향으로만 처리)**.
- ⭐ SCAN 계산 문제는 지문의 직전 이동으로 **현재 진행 방향부터** 확정한다. 방향 유지가 원칙 — 70→60이면 감소 방향.
- ⭐ 기아(무한 대기) 계보: **SSTF에서 발생** → SCAN이 완화 → **N-Step SCAN이 확실히 방지**. 에센바흐는 회전 지연까지 고려하는 유일한 기법.

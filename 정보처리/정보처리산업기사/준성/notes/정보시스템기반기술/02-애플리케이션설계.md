# 02. 애플리케이션설계

> 출처: `_공통/출제기준/정보시스템기반기술.md`의 주요항목 순번을 따른다.

## 공통 모듈 설계

### 핵심 개념
- 정의: UML(Unified Modeling Language) — 소프트웨어 시스템의 구조와 동작을 **명세화·시각화·구축·문서화**하기 위한 **표준화된 범용 모델링 언어**. 건축 설계도처럼, 개발자와 이해관계자가 시스템을 두고 의사소통하기 위한 표준 도면(다이어그램)이다.
- 왜 중요한가: 복잡한 시스템을 객체지향 관점에서 쉽게 모델링·공유할 수 있다. 구성 요소(사물·관계·다이어그램)와 관계의 종류별 정의를 바꿔치기해 "옳지 않은 것"을 고르게 하는 문제가 기출에 나온다.

### 상세 정리

**UML의 3대 구성 요소: 사물 · 관계 · 다이어그램**

| 구성 요소 | 설명 |
|-----------|------|
| 사물 (Things) | 모델을 구성하는 가장 기본적·핵심적인 요소. 도형으로 표현 |
| 관계 (Relationships) | 사물과 사물이 어떻게 연결·연관되는지를 선으로 표현 |
| 다이어그램 (Diagrams) | 사물과 관계를 모아 알아보기 쉽게 그린 도면 |

**사물(Things)의 종류**

- **구조 사물**: 클래스·컴포넌트처럼 시스템의 개념적·물리적 요소
- **행동 사물**: 상호작용, 상태 변화 등 동적 행위
- **그룹 사물**: 패키지처럼 요소들을 묶어주는 것
- **주해(메모) 사물**: 부가 설명·제약조건을 적는 것

**관계(Relationships)의 종류** — 기출 최다 출제 지점

| 관계 | 정의 |
|------|------|
| 연관 (Association) | 2개 이상의 사물이 서로 관련되어 있음을 표현 |
| 의존 (Dependency) | 한 사물이 다른 사물에 영향을 주는 **짧은 시간 동안**의 관계 |
| 일반화 (Generalization) | 상속처럼 일반적인 것(부모)과 구체적인 것(자식)의 관계 |
| 실체화 (Realization) | 사물이 할 수 있거나 해야 하는 기능(인터페이스)을 구체적으로 구현·그룹화하는 관계 |
| 포함/집합 (Composition/Aggregation) | **하나의 사물이 다른 사물에 포함되어 있는** 전체-부분 관계 |

- 함정 포인트: "하나의 사물이 다른 사물에 **포함**되어 있는 관계"는 **의존이 아니라 포함(Composition)·집합(Aggregation)**에 대한 설명이다. 의존은 포함이 아니라 짧은 시간의 영향 관계.

**다이어그램(Diagrams)의 구분**

- **구조적(정적) 다이어그램**: 시스템의 뼈대·정적 모습 — 예: 클래스 다이어그램, 컴포넌트 다이어그램
- **행위적(동적) 다이어그램**: 시스템의 동적 흐름 — 예: 유스케이스 다이어그램, 시퀀스 다이어그램

### 암기 포인트
- ⭐ UML 구성 3요소 = **사물·관계·다이어그램**. 관계 정의 문제에서 "포함"이라는 말이 나오면 의존이 아니라 **포함/집합** 관계인지 의심하라.

# 04. 애플리케이션테스트관리

> 출처: `_공통/출제기준/소프트웨어개발.md`의 주요항목 순번을 따른다.

## 애플리케이션 테스트케이스 설계

### 핵심 개념
- **테스트 = 오류 발견**, **디버그 = 오류 수정**. 테스트 원리: **파레토 법칙(80/20)** = 오류의 80%는 전체 모듈의 20%에서 발생.
- **검증(Verification)** = "제품을 올바르게 만들고 있는가"(개발자 관점, 각 단계 산출물 검토). **확인(Validation)** = "올바른 제품을 만들고 있는가"(사용자 요구 충족).
- **화이트박스 테스트(구조/원시코드 기반)**: 프로그램 내부 논리(구조)를 보고 테스트. 기법: **구문(문장) 커버리지, 결정(분기) 커버리지, 조건 커버리지, 조건/결정 커버리지, 다중 조건 커버리지, 기본 경로(Basis Path)**. **McCabe의 Cyclomatic Complexity** = 제어흐름 그래프의 복잡도(=에지−노드+2, 또는 영역 수)로 독립 경로 수 산정.
- **블랙박스 테스트(기능/입출력 기반)**: 내부 구조 모르고 기능·입출력만으로 테스트. 기법: **동등(동치) 분할, 경계값 분석, 원인-효과 그래프, 비교 테스트, 결정 테이블, 상태 전이**.
- **테스트 오라클(Oracle)**: 테스트 결과가 참/거짓인지 판정할 **사전에 정의된 기대값(참값)**. 종류: **참(True) 오라클**(모든 입력에 기대값), **샘플링 오라클**(특정 입력만), **휴리스틱(추정) 오라클**(나머지는 추정), **일관성 검사 오라클**(변경 전후 비교).
- **테스트 레벨/종류**: 단위→통합→시스템→인수 테스트. **알파 테스트**=개발자 장소에서 사용자가 개발자 앞에서 검사(통제 환경). **베타 테스트**=사용자 환경에서 사용자가 직접. **강도(스트레스) 테스트**=과부하 시 정상 동작 확인.
- **NS Chart(Nassi-Schneiderman)**: 구조적 프로그래밍용 설계도(GOTO 없음), 화이트박스와 연관. **테스트 케이스 자동 생성 도구**: 동치분할/경계값/원인효과/결정테이블 기반.

### 암기 포인트
- ⭐ **화이트박스=내부 구조**(**커버리지·경로**), **블랙박스=기능/입출력**(**동등분할·경계값**).
- ⭐ McCabe 복잡도 = **V(G)=E−N+2** = 독립 경로 수.
- ⭐ 알파=**개발자 앞**(통제환경), 베타=**사용자 환경**.
- ⭐ 테스트 오라클 = **미리 정의된 기대값(참값)**.

## 애플리케이션 통합 테스트

### 핵심 개념
- **단위 테스트(Unit Test)**: 개별 모듈이 정확히 구현됐는지, 예정 기능을 수행하는지 점검. 모듈 내부 오류(로직·자료구조·경계값) 발견이 목적.
- **통합 테스트(하향식/상향식)**: 모듈 간 인터페이스·결합 검증.
  - **하향식(Top-Down)**: 상위→하위. 하위 미완성 모듈 대신 **Stub(스텁)**=일시적 가짜 모듈(조건만 제공) 사용.
  - **상향식(Bottom-Up)**: 하위→상위. 상위 미완성 모듈 대신 **Driver(드라이버)**=하위 모듈 호출·파라미터 전달하는 가상 모듈 사용.
- **테스트 하네스(Harness)**: 단위/통합 테스트 지원 환경. 구성: **테스트 드라이버, 테스트 스텁, 테스트 슈트(케이스 집합), 테스트 케이스, 테스트 스크립트, 테스트 리포트**.
- **테스트 자동화 도구**: 성능 테스트(가상 사용자 생성→처리량·응답시간·자원사용률 측정), 결함 추적 도구 등.

### 암기 포인트
- ⭐ 하향식=**Stub(스텁)**, 상향식=**Driver(드라이버)**. (위=스텁, 아래=드라이버)
- ⭐ 하네스 = 테스트 **지원 환경**(드라이버+스텁+슈트+스크립트).

## 애플리케이션 성능 개선

### 핵심 개념
- **알고리즘**: 정렬·검색·설계 기법의 효율(시간/공간 복잡도)을 다룬다.
  - **정렬**: 버블(O(n²)), 선택(O(n²)), 삽입(O(n²)), **퀵 정렬(분할정복+피봇, 평균 O(nlog₂n)/최악 O(n²)=n(n−1)/2)**.
  - **검색**: **이진 검색(Binary Search)**=정렬된 배열에서 중앙값 비교로 반씩 줄임(O(log₂n)), **반드시 정렬 필요**.
  - **시간 복잡도**: O(1)=입력크기 무관(상수), O(log n), O(n), O(nlog n), O(n²). **분할 정복(Divide and Conquer)**=문제를 나눠 해결(퀵/병합 정렬). 설계 기법: 분할정복, 동적계획법(DP), 탐욕(Greedy), 백트래킹.
- **소스코드 품질분석 도구**: **정적 분석(Static)**=실행 없이 소스 코드 분석(코딩 표준·결함·복잡도 점검: SonarQube, PMD, FindBugs, Checkstyle, Cppcheck). **동적 분석(Dynamic)**=실행하며 분석(메모리 누수·성능).
- **코드 품질·리뷰**: **클린 코드 작성 원칙**(가독성, 단순성, 의존성 최소, 중복 최소, 일관성, 정확성). **외계인 코드(Alien Code)**=이해 어려운 방대한 레거시 코드. **인스펙션(Inspection)**·**워크스루(Walkthrough)**·**FTR(정형 기술 검토)**=공식/비공식 코드·산출물 검토 회의로 결함 사전 발견.

### 암기 포인트
- ⭐ 퀵 정렬 = **분할정복 + 피봇**, 최악 **n(n−1)/2**회 비교. 이진 검색 = **정렬 필수**, O(log₂n).
- ⭐ O(1)=**입력 무관**(해시 조회 등).
- ⭐ 정적 분석 = **실행 없이**(SonarQube/PMD/FindBugs), 동적 분석 = **실행하며**.
- ⭐ 클린코드 원칙 = **가독성·단순성·중복↓·일관성**.

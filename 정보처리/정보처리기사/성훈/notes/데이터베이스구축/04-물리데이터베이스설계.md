# 04. 물리데이터베이스설계

> 출처: `_공통/출제기준/데이터베이스구축.md`의 주요항목 순번을 따른다.
> 세부항목: 물리요소 조사 분석(분산 DB) · 물리속성 설계(파티셔닝·물리적 설계) · 물리DB 모델링(무결성·키 종류·반정규화) · 품질검토(CRUD 분석)

## 물리요소 조사 분석 — 분산 데이터베이스

### 핵심 개념
- **분산 데이터베이스(Distributed DB)**: 논리적으로는 하나의 시스템, 물리적으로는 여러 컴퓨터 사이트에 분산된 DB.
- **목표(투명성 Transparency)**: **위치 투명성**, **중복 투명성**, **병행 투명성**, **장애 투명성**. (Media Access Transparency는 해당하지 않음)
- **장점**: 지역 자치성 높음, 분산 제어 가능, 효용성·융통성 높음, 확장 용이.
- **단점**: 설계가 어렵고 비용 증가, **잠재적 오류 증가**, 보안·무결성 관리 어려움.
- **주요 구성 요소**: 분산 처리기, 통신 네트워크. (P2P 시스템·단일 데이터베이스는 아님)
- 분산 투명성은 **사용자가 분산을 인식하지 못하게** 하는 것(배제하는 것이 아님).

### 암기 포인트
- ⭐ 투명성 4종 = **위치·중복·병행·장애** (Media Access Transparency는 아님).
- ⭐ 장애 투명성 = 장애 발생 시에도 **데이터 무결성 보장**.
- ⭐ 단점 = 설계 어려움·비용 증가·**잠재적 오류 증가**(감소가 아님).
- ⭐ 분산 투명성은 **인식 못하게**(배제가 아님).

- 🔁 기출 2023-2회-필기 #46: 분산 DB 시스템 — 투명성 배제해야 한다는 것은 틀림
- 🔁 기출 2023-2회-필기 #56: 분산 DB 투명성이 아닌 것(Media Access Transparency)
- 🔁 기출 2023-3회-필기 #56: 분산 DB 시스템 — 구성 요소 설명 틀림(P2P·단일 DB 아님)
- 🔁 기출 2024-2회-필기 #52: 장애 투명성 — 장애 발생 시 데이터 무결성 보장

## 데이터베이스 물리속성 설계 — 물리적 설계 단계

### 핵심 개념
- **물리적 설계**: 효율적인 방법으로 데이터를 저장하는 것. 처리량·응답시간·디스크 용량 고려.
- 수행 사항: **저장 레코드 양식 설계, 레코드 집중의 분석 및 설계, 접근 경로 설계**.
- 목표 DBMS에 맞는 **스키마 설계**는 논리적 설계 단계의 작업(물리적 설계가 아님).
- **트랜잭션 인터페이스 설계**도 논리적 설계 단계(물리적 설계가 아님).

### 암기 포인트
- ⭐ 물리적 설계 = **저장 레코드 양식·레코드 집중·접근 경로** 설계.
- ⭐ 스키마 설계·트랜잭션 인터페이스 = **논리적 설계** (물리적 설계가 아님).

### 기출 연계

- 🔁 기출 2024-3회-필기 #51: 물리적 설계 단계 식별(저장 레코드 양식·집중·접근 경로)
- 🔁 기출 2025-2회-필기 #48: 물리적 설계 단계 사항이 아닌 것(스키마 설계)

## 데이터베이스 물리속성 설계 — 파티셔닝

### 핵심 개념
- **파티션(Partition)**: 하나의 테이블을 작은 논리적 단위로 나눈 것. 물리적으로 별도 세그먼트에 저장.
- **수평 분할** 기법: **라운드로빈**, **범위 분할**, **해시 분할**. (예측 분할은 아님)
- 장점: 쿼리 성능 향상, 데이터 가용성 향상.

### 암기 포인트
- ⭐ 수평 분할 기법 = **라운드로빈·범위·해시** (예측 분할은 아님).

### 기출 연계
- 🔁 기출 2023-2회-필기 #45: 수평 분할 기법이 아닌 것(예측 분할)
- 🔁 기출 2024-2회-필기 #50: 파티션 설명 — 백업·복구 속도 느려짐은 틀림

## 물리 데이터베이스 모델링 — 데이터베이스 무결성

### 핵심 개념
- **개체 무결성(Entity Integrity)**: 기본키를 구성하는 속성은 **널(Null) 값이나 중복 값을 가질 수 없음**.
- **참조 무결성(Referential Integrity)**: 외래키는 참조하는 릴레이션의 기본키 값과 동일해야 함. 외래키를 변경하면 이를 참조하는 릴레이션의 기본키도 변경해야 함.
- **CASCADE**: 부모 테이블 항목 삭제 시 자동으로 자식 테이블의 해당 레코드도 삭제(참조 무결성 유지).
- 무결성 규정: 데이터가 만족해야 할 제약 조건. 도메인·키·종속성 등이 대상. (불법 사용자 보호는 **보안**의 영역, 무결성이 아님)

### 암기 포인트
- ⭐ 개체 무결성 = **기본키 null·중복 불가** (가장 빈출).
- ⭐ 참조 무결성 = 외래키 ↔ 기본키 **동일**. **CASCADE** = 자식 자동 삭제.
- ⭐ 무결성 ≠ 보안 (불법 사용자 보호는 보안).

### 기출 연계

- 🔁 기출 2024-1회-필기 #45: 개체 무결성(기본키 null 아닌 원자값)
- 🔁 기출 2024-2회-필기 #46: 참조 무결성(외래키 변경 시 참조 관계 유지)
- 🔁 기출 2024-3회-필기 #41: CASCADE 옵션(참조 무결성, 자식 레코드 자동 삭제)
- 🔁 기출 2024-3회-필기 #45: 개체 무결성(기본키 null 아닌 원자값)

## 물리 데이터베이스 모델링 — 키 종류

### 핵심 개념
- **슈퍼키(Super Key)**: 유일성 **O**, 최소성 **X** (한 개 이상의 속성 집합).
- **후보키(Candidate Key)**: 유일성 **O**, 최소성 **O** (튜플을 고유하게 식별).
- **기본키(Primary Key)**: 후보키 중 선택. null·중복 불가. 검색 시 **반드시 필요한 것은 아님**.
- **대체키(Alternate Key)**: **후보키** 중 기본키를 제외한 나머지 (슈퍼키 중이 아님).
- **외래키(Foreign Key)**: 다른 릴레이션의 **기본키를 참조**하는 속성. 릴레이션 간 참조 관계 표현 도구.

### 암기 포인트
- ⭐ **슈퍼키**(유일O 최소X) ⊃ **후보키**(유일O 최소O) → **기본키**(선택) + **대체키**(나머지).
- ⭐ **대체키** = 후보키 중 기본키 제외 (**슈퍼키** 중이 아님).
- ⭐ **외래키** = 다른 릴레이션의 **기본키를 참조**.
- ⭐ 기본키는 검색 시 **반드시 필요하지 않음**(인덱스 없이도 검색 가능).

### 기출 연계
- 🔁 기출 2023-1회-필기 #51: 기본키 속성 설명 — 검색 시 반드시 필요한 것은 아님
- 🔁 기출 2023-2회-필기 #58: 외래키 식별(과목번호)
- 🔁 기출 2023-3회-필기 #59: 키 종류 설명 — 대체키는 후보키 중 기본키 제외(슈퍼키가 아님)
- 🔁 기출 2024-1회-필기 #55: 슈퍼키(유일성 O, 최소성 X)

- 🔁 기출 2024-3회-필기 #59: 외래키(다른 릴레이션의 기본키를 참조)

- 🔁 기출 2025-3회-필기 #57: 슈퍼키(유일성 O, 최소성 X)

## 물리 데이터베이스 모델링 — 반정규화

### 핵심 개념
- **반정규화(Denormalization)**: 정규화된 엔티티·속성·관계를 **시스템 성능 향상과 개발·운영의 단순화**를 위해 **중복·통합·분리** 등을 수행하는 데이터 모델링 기법.
- 정규화로 인한 **조인(Join) 과다로 조회 성능 저하** 시 적용. 읽기 성능 향상이 주된 목적.
- 기법: **테이블 통합·분할**, **중복 속성 추가**, **파생 속성 추가** 등. 데이터 무결성 저하 위험을 감수하므로 신중 적용.
- 물리적 설계 단계에서 수행(논리적 설계의 정규화 이후).

### 암기 포인트
- ⭐ 반정규화 = 정규화의 역방향, **성능 향상이 목적**, 중복·통합·분리 수행.

### 기출 연계
- 🔁 기출 2023-3회-필기 #49: 반정규화의 개념(성능 향상을 위해 중복·통합·분리)
- 🔁 기출 2025-3회-필기 #50: 반정규화의 개념(성능 향상을 위해 중복·통합·분리)

## 물리데이터 모델 품질검토 — CRUD 분석

### 핵심 개념
- **CRUD 분석**: 데이터베이스에 영향을 주는 **C**reate(생성), **R**ead(읽기), **U**pdate(갱신), **D**elete(삭제) 연산으로 프로세스와 테이블 간에 **매트릭스를 만들어 트랜잭션을 분석**하는 기법.
- **프로세스-테이블 간 매트릭스** 상관관계를 점검하여 누락된 테이블·프로세스를 식별하고 데이터 무결성을 검증.
- 각 프로세스가 테이블에 어떤 연산을 수행하는지 매핑 → 미사용 테이블·과부하 테이블 등 설계 오류를 조기 발견.

### 암기 포인트
- ⭐ CRUD = **Create·Read·Update·Delete**, 프로세스-테이블 **매트릭스**로 트랜잭션 분석.

### 기출 연계

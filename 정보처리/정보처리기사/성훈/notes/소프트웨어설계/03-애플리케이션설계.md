# 03. 애플리케이션설계

> 출처: `_공통/출제기준/소프트웨어설계.md`의 주요항목 순번을 따른다.
> 세세항목: 설계 모델링 · 소프트웨어 아키텍처 / 객체지향(OOP) · 디자인패턴

## 공통 모듈 설계

### 핵심 개념
- **모듈(Module)**: 모듈화로 분리된 시스템의 각 기능 단위(서브루틴·서브시스템·프로그램·작업 단위). **컴포넌트(Component)**는 인터페이스로만 접근 가능한 독립적 재사용 단위.
- **응집도(Cohesion)**: 모듈 **내부** 구성요소 간의 기능적 연관 정도. **높을수록 좋다**. 결합도(Coupling): 모듈 **간**의 의존·상호작용 정도. **낮을수록 좋다**. → "결합도↓, 응집도↑"이 바람직한 설계.
- **결합도 단계(약→강)**: 자료(Data) → 스탬프(Stamp) → 제어(Control) → 외부(External) → 공통(Common) → 내용(Content). 가장 강한 것 = 내용 결합도.
- **추상화(Abstraction) 3종**: 과정(자료 흐름) · 자료(데이터) · 제어 추상화. ("강도 추상화"는 없음.)
- **바람직한 설계 지침**: 결합도↓·응집도↑, 복잡도·중복↓, 일관성 유지, 적절한 모듈 크기, 단일 입출구. "결합도 강할수록 좋다"·"모듈을 극단적으로 작게 해 병행성↑"은 모두 틀림.

### 상세 정리

**결합도 종류(약한 순)**
| 결합도 | 의미 |
| :--- | :--- |
| 자료(Data) | 매개변수로 **단순 자료** 전달 (가장 약=가장 좋음) |
| 스탬프(Stamp) | 자료구조(배열·레코드) 형태 전달 |
| 제어(Control) | 제어 신호(플래그) 전달 |
| 공통(Common) | 전역 데이터 공유 |
| 내용(Content) | 다른 모듈의 내부 내용 직접 참조 (가장 강=가장 나쁨) |

**하향식 vs 상향식 설계**
- **하향식(Top-down)**: 상위→하위. 인터페이스가 먼저 정의돼 **통합이 간단**. 초기에 저수준 데이터 구조 세부사항 불필요.
- **상향식(Bottom-up)**: 하위 모듈부터 설계 후 결합. 인터페이스 미정 상태에서 기능 추가가 어려움.

**fan-in / fan-out**: fan-in = 해당 모듈을 **호출하는** 상위 모듈 수(클수록 재사용성↑), fan-out = 해당 모듔이 **호출하는** 하위 모듈 수.

**코드 설계**: 순차 코드(일련번호) · 블록 코드 · 연상 코드 · 표의 숫자 코드(물리적 수치 그대로) · 10진 코드 등.

**HIPO(Hierarchy Input Process Output)**: **하향식** 소프트웨어 개발을 위한 문서화 도구. 가시적 도표(Visual Table of Contents) · 총체적 도표(Overview Diagram) · 세부적 도표(Detail Diagram)로 구성. 기능과 자료의 의존관계 동시 표현. ("상향식 도구"라는 설명은 항상 틀림.)

**N-S(Nassi-Schneiderman) Chart**: 논리 기술 중심의 **도형식** 표현. 연속·선택·다중선택·반복 제어 구조. **화살표를 사용하지 않음**(flowchart와의 차이).

**공통 모듈 명세 원칙**: 정확성 · 명확성(한 가지로만 해석) · 완전성 · 일관성 · 추적성 · 독립성.

### 암기 포인트
- ⭐ **결합도↓, 응집도↑**. 결합도 약→강 = **자·스·제·외·공·내**(자료→스탬프→제어→외부→공통→내용).
- ⭐ HIPO = **하향식** 도구, 3종 도표(가시적·총체적·세부적). N-S 차트 = **화살표 없는** 도형식.
- ⭐ 추상화 3종 = 과정·자료·제어. 코드 설계에서 "일련번호"=순차 코드, "물리적 수치"=표의 숫자 코드.

## 객체지향 설계

### 핵심 개념
- **객체지향(OOP) 핵심 개념**: 클래스(유사 객체들의 공통 특성을 묶은 추상화) · 객체(속성+메소드) · 캡슐화(데이터+메소드 묶기, **정보은닉**과 밀접) · 상속(상위 클래스 속성·연산 전달) · 다형성(동일 메시지로 다른 동작) · 메시지(객체에게 행위 지시 명령) · 메소드(객체의 구체적 연산).
- **클래스 구성**: 속성(Attribute = 상태·정보) + 오퍼레이션/메소드(동작·행위).
- **객체지향 연관성**: 연관화(참조) · 집단화(부분-전체) · 일반화(상속, is-a) · 분류화(classification, is instance of).

### 상세 정리

**SOLID 원칙 (객체지향 설계 5원칙)**
| 약자 | 원칙 | 핵심 |
| :--- | :--- | :--- |
| SRP | 단일 책임 원칙 | 객체는 하나의 책임만 |
| OCP | 개방-폐쇄 원칙 | 확장엔 열리고 변경엔 닫힘 |
| LSP | 리스코프 치환 원칙 | 자식은 부모를 대체 가능 |
| ISP | 인터페이스 분리 원칙 | 사용하지 않는 인터페이스에 의존 금지 |
| DIP | 의존 역전 원칙 | 추상성 높은 클래스에 의존 |

> SSO(Single Sign On)는 SOLID가 아님. "ISP = 단일 책임"은 틀린 설명(SRP가 단일 책임).

### 암기 포인트
- ⭐ 캡슐화 ↔ **정보은닉** 짝꿍. 클래스 = 속성(상태) + 오퍼레이션(동작).
- ⭐ SOLID 다섯 글자 = **S·O·L·I·D**(SRP·OCP·LSP·ISP·DIP). SSO는 제외.
- ⭐ ISP(인터페이스 분리) = "사용 안 하는 메소드와 의존관계 맺지 마라".

### 핵심 개념 (디자인패턴)
- **디자인 패턴**: 모듈의 세분화된 역할·인터페이스 구현 시 참조할 수 있는 **전형적 해결 방식·예제**. 객체지향 설계 생산성 ↑, 재사용 시간 단축, 구조 파악 용이.
- **GoF 분류 3종**: 생성(Creational - Singleton·Factory·Prototype·Builder) · 구조(Structural - Adapter·Bridge·Composite·Decorator·Proxy) · 행위(Behavioral - Observer·State·Mediator). ("객체 패턴"이라는 분류는 없음.)
- **패턴 구성 요소**: 패턴 이름 · 문제 및 배경 · 해결 방법 · 실제 적용 사례 · 재사용 가능 샘플코드. ("개발자 성명"은 제외.)

### 암기 포인트 (디자인패턴)
- ⭐ GoF 3분류 = **생성·구조·행위**. "객체 패턴"은 오답.
- ⭐ **Builder = 생성 패턴**(구조 아님), **Observer/State = 행위 패턴**(구조 아님). 이 둘이 "구조 패턴이 아닌 것" 문제의 단골 정답.
- ⭐ 디자인 패턴은 "절차형 언어"와 함께 쓸 때 효용이 극대화된다는 설명은 **틀림**(객체지향 전용).

## 셀프 체크
- [ ] 결합도 단계(약→강)와 응집도의 방향(↑↓)을 암기했는가
- [ ] HIPO(하향식, 3종 도표)·N-S 차트(화살표 없음)의 핵심을 말할 수 있는가
- [ ] SOLID 5원칙을 암기하고, ISP가 무엇인지 설명할 수 있는가
- [ ] GoF 3분류(생성·구조·행위)와 대표 패턴을 구분할 수 있는가

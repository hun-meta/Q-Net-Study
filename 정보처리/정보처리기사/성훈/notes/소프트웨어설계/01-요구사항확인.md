## 요구사항 확인

> 세세항목: 요구분석 기법 · UML · 애자일(Agile). 아래는 그중 **소프트웨어 생명주기(SDLC) 모형 / 프로토타입** 중점 정리. ⭐(2025-3회-필기 #1 오답 → 중점 보강)

### 핵심 개념
- 정의(SDLC 모형): **소프트웨어 생명주기(SDLC, Software Development Life Cycle) 모형**은 개발 계획부터 분석·설계·구현·테스트·유지보수·폐기까지의 전 과정을 체계화한 **소프트웨어 개발 프로세스 모형**이다. 프로토타입은 이 SDLC 모형의 한 종류(상위 개념 = SDLC 모형).
- 정의(프로토타입): 개발 초기에 사용자 요구사항을 정확히 파악하기 위해 핵심 기능·인터페이스 중심의 **견본품(시제품/초기 모형)**을 빠르게 제작한 뒤, 사용자 피드백을 수렴해 요구사항을 지속적으로 수정·보완해 나가는 개발 모형.
- 왜 중요한가: 각 SDLC 모형의 **특징·진행 순서·장단점 차이**가 반복 출제된다. 프로토타입은 특히 "요구사항을 명확히 하기 위한 도구"라는 성격과 폐기형/진화형 구분이 자주 나온다.

### 상세 정리

**프로토타입 모형의 특징**
- 발주자(사용자)와 개발자 간의 **언어적·시각적 이해 차이(의사소통 갭)**를 줄여 준다.
- 유형 2가지:
  - **폐기형(Throwaway)**: 요구사항 수집·확인 후 프로토타입을 **버리고** 본 개발에 착수.
  - **진화형(Evolutionary)**: 프로토타입을 계속 보완하여 **최종 시스템으로 발전**시킴.

**동일 레벨(다른 SDLC 개발 모형)과의 비교** — 시험은 "차이점"을 묻는다.

| 모형 | 핵심 특징 | 프로토타입과의 차이 |
| :--- | :--- | :--- |
| **폭포수(Waterfall)** | 타당성검토→분석→설계→구현→테스트→유지보수 **순차** 진행 | 순차·고정 구조라 개발 중간의 요구사항 변경·피드백 반영이 매우 어려움 |
| **프로토타입(Prototype)** | 견본품 제작 → 사용자 피드백 → 요구사항 정제 반복 | (기준 모형) 요구사항 명확화에 초점 |
| **나선형(Spiral)** | 계획 → **위험 분석(Risk Analysis)** → 개발/검증 → 고객 평가를 **반복** | 프로토타입 장점 + **위험 분석** 강화 → 대규모·고위험 프로젝트에 적합 |
| **애자일(Agile)** | 짧은 주기(Sprint)로 개발-배포 반복, 유연 대응 | 프로토타입은 요구 명확화 "도구", 애자일은 개발 체계 전체가 가볍고 유연한 "방법론" |
| **V-모형(V-Model)** | 폭포수 기반, 각 개발 단계에 상응하는 테스트(단위/통합/시스템/인수)를 1:1 매핑 | 검증(Verification)·확인(Validation) 테스트의 체계적 관리에 초점 |

**프로토타입의 목적(4가지)**
1. **요구사항 구체화·검증**: 사용자가 명확히 표현 못한 추상적 요구를 시각적 모형으로 정의·정제.
2. **의사소통 갭 해소**: 결과물에 대한 사용자–개발자 인지 차이를 극초반에 축소 → 요구 불일치 실패 방지.
3. **재작업 비용·개발 위험 감소**: 후반부에 발견되면 막대한 비용이 드는 요구 변경/오류를 초반에 발견해 수정 비용 최소화.
4. **타당성·UI/UX 사전 평가**: UI 편의성, 핵심 알고리즘/기능의 기술적 구현 타당성을 본 개발 전에 검증.

### 암기 포인트
- ⭐ 프로토타입 = **"견본품으로 요구사항 명확화"** / 유형은 **폐**기형·**진**화형(폐진).
- ⭐ 상위 개념은 **SDLC 모형**, 형제 개념은 **폭포수·나선형·애자일·V모형**.
- ⭐ **나선형 = 프로토타입 + 위험 분석**(Risk Analysis가 키워드면 나선형).
- ⭐ **위험 분석 반복**은 나선형, **테스트 1:1 매핑**은 V-모형, **순차·변경취약**은 폭포수.

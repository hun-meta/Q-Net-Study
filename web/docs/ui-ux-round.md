## 프롬프트 본문

Q-Net 기출 풀이 로컬 웹 앱(`web/`)의 UI/UX 전면 개선 작업이다. **기능 로직·서버 API는 완성·검증 상태이므로 건드리지 말고**, 그 위의 화면 구성과 경험을 재설계하라.

### 최우선 UX 목표 (이 순서로 판단)
1. **직관성 — 첫 화면에서 앱이 뭘 해주는지 한눈에 파악**: 처음 연 사람이 설명 없이 "기출을 풀고 → 채점받고 → 오답을 복습하는 앱"임을 3초 안에 인지하고, 다음 행동(이어풀기/새 기출 풀기/업로드/개념 보기)이 클릭 1~2번 거리에 있어야 한다.
2. **작업 중심 흐름**: 화면이 "파일 구조"가 아니라 "사용자가 하려는 일"을 따라 조직되어야 한다.
3. **학습 현황의 상시 가시성**: 내 진행 상태(최근 점수·오답 수·찍음 수·추이)가 홈에서 바로 보인다.

### 🔓 화면 구성 재설계 권한 (v2 핵심)
- **정보 구조(IA)·화면 배치·네비게이션·진입 플로우를 자유롭게 바꿔도 된다.** index.html 구조, app.js의 뷰 전환/마운트 방식, 각 렌더 함수의 배치·조합을 재구성할 수 있다.
- 아래 "유지 계약"만 지키면 화면을 통째로 재편해도 된다:
  - 서버 API 계약(경로·메서드·요청/응답 스키마) — 변경 금지, 새 API 추가 요청도 금지(프론트로 해결)
  - 커스텀 이벤트 계약: `qnet:attempt-submitted`(detail=제출 응답), `qnet:fs-change` — 발행·수신 의미 유지
  - 데이터 로직(채점 표시 값, draft 저장 주기, 컨텍스트 조립 내용) — 표현만 바꾸고 계산은 유지
  - X-QNet-Token 첨부(store.js apiFetch 경유) 유지
- 기존 파일 분할(js/*.js)은 재구성 가능하되, 기능 단위 모듈성은 유지하라(한 파일로 뭉치지 말 것).

### 화면 구성 제안 (참고안 — 더 나은 안이 있으면 그것으로, 단 채택 이유를 설계 문서에 남겨라)
- **홈 = 학습 대시보드**: 상단에 "이어풀기" CTA(진행 중 드래프트가 있으면 최우선 노출) → 내 현황 카드(최근 시도 점수·합격선 대비·오답/찍음 수·추이 미니 차트) → 자격증 카드 그리드(자격증별 진행 상태 뱃지) → 하단에 "새 자격증 등록 / 기출 업로드" 보조 동선.
- **풀이 화면 = 몰입 모드**: 헤더 최소화, 좌 PDF 뷰어(넓게) + 우 OMR 레일(고정 스크롤) + 하단/우측 슬라이드 도구 패널(개념·챗) — 풀이 중 방해 요소 제거, 경과시간·진행률 상시 표시.
- **결과 화면 = 행동 유도**: 채점 히어로 → "오답 키워드 입력 → 개념 보기 → 챗으로 질문" 흐름을 다음 단계 버튼으로 연결(오답 복습 퍼널).
- **온보딩**: 닉네임 미설정 시 전체 화면 웰컴(앱 소개 한 줄 + 닉네임 드롭다운/등록) — 완료 후 대시보드 진입.
- 네비게이션: 사이드바 트리(종류>자격증) 대신 **대시보드 카드 → 자격증 상세(기출 목록+업로드) → 풀이** 의 단계형 드릴다운 권장. 상단 브레드크럼으로 복귀 동선.

### 하드 제약 (위반 금지)
1. 번들러·프레임워크·전처리기 도입 금지 (무빌드 바닐라 ES 모듈 유지).
2. 외부 CDN·웹폰트·원격 리소스 금지 (로컬 전용) — 시스템 폰트 스택, 아이콘은 유니코드/인라인 SVG.
3. 서버 코드(web/server/**) 수정 금지.
4. `cd web && npm test` 전체 통과 유지, 브라우저 콘솔 에러 0.
   - 참고: 프론트 파일을 참조하는 테스트가 있으면(예: 모듈 임포트 그래프 검사) 재구성에 맞춰 테스트를 갱신하는 것은 허용.

### 디자인 시스템 요구
- CSS 변수 체계: 색(서페이스·텍스트 3단계·테두리), 상태색 4종(성공/경고/위험/정보), 타이포 스케일 4단계, 간격 스케일, radius, shadow.
- 라이트/다크: `prefers-color-scheme` 기본 + 토글(localStorage), 두 모드 모두 명도 대비 4.5:1.
- 한국어 가독성: `-apple-system, "Apple SD Gothic Neo", "Malgun Gothic", sans-serif`, 본문 줄간격 1.6.

### 컴포넌트 품질 기준 (재배치는 자유, 각 컴포넌트가 갖출 것)
- **기출 카드/행**: 연도·회차·구분 뱃지 + 상태 뱃지(채점가능/열람만/정답미등록) + 내 최근 점수.
- **OMR**: 과목 구분 헤더, 선택 상태가 명확한 라디오, 찍음 토글의 뚜렷한 시각 구분, 진행률 바(응답/전체), 경과시간, 키보드(1~4 답, g 찍음, ↑↓ 문항 이동 — input 포커스 시 비활성).
- **채점 결과**: 총점 히어로+판정 뱃지, 과목별 게이지(과락 기준선 40 표시·위험색), 시도 추이 인라인 SVG 차트, 확신 정답률 강조.
- **키워드 폼**: X·찍음 문항 인라인 입력 + 건너뛰기.
- **개념 패널**: md 타이포(제목·표·코드), 내/타인/공유 해설 구분(작성자 뱃지).
- **챗**: 말풍선, 스트리밍 커서, 문항 컨텍스트 칩, 승인 다이얼로그(목적지 체크박스+설명).
- **업로드**: 드롭존(드래그오버 상태), 단계 진행 표시(업로드→판독→검증→등록), 실패 시 수동 정답 폼.
- **삭제 확인 모달**: 삭제 대상 경로·파일 수 목록, 닉네임 입력 확인, 위험 스타일, 포커스 트랩+ESC.
- **알림**: 토스트(정보/성공) + 상단 배너(감사 경고=경고색, CLI 미설치=중립). fs-change 자동 갱신은 무음.
- **빈 상태/로딩/에러**: 행동 유도 버튼이 있는 빈 상태, 스켈레톤 로딩, 사람이 읽는 에러 문구(409/503).

### 접근성
포커스 링 유지, 모달 포커스 트랩+ESC, aria-label(아이콘 버튼), 클릭 타깃 ≥32px, 키보드만으로 "대시보드→풀이→제출→결과" 완주 가능.

### 진행 방식 (2단계)
1. **설계 먼저**: 재구성한 화면 구성안(화면 목록·와이어프레임 텍스트/ASCII·네비게이션 흐름도·참고안 대비 변경 이유)을 `web/docs/ui-design.md`로 작성해 제출.
2. **구현**: 설계안대로 구현 후 검증.

### 검증 (완료 조건)
1. `cd web && npm test` 전체 통과 (기능 회귀 0).
2. Playwright 스크린샷: 온보딩·대시보드·자격증 상세·풀이·결과·개념/챗 — 라이트/다크 각각, 콘솔 에러 0.
3. 키보드만으로 풀이→제출 플로우 완주 확인.
4. 첫 화면 3초 테스트: 스크린샷만 보고 "무엇을 하는 앱인지, 지금 뭘 누르면 되는지"가 설명 없이 드러나는지 자체 평가를 설계 문서에 기록.

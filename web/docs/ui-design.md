# Q-Net 기출 풀이 웹앱 — 화면 구성 설계안 (UI/UX 라운드 v2)

작성: 메인 에이전트(설계 담당). 근거 프롬프트: `.omc/prompts/ui-ux-round.md` (v2).
선행 라운드: 스타일 현대화(디자인 토큰·다크 모드·컴포넌트 스타일)는 적용 완료 상태이며, 본 설계는 그 위에 **정보 구조(IA)·화면 배치·네비게이션을 재편**한다.

## 1. UX 목표와 판단 기준

1. **첫 화면 3초 파악**: 처음 연 사람이 "기출을 풀고 → 채점받고 → 오답을 복습하는 앱"임을 즉시 인지. 다음 행동(이어풀기/새 기출/업로드)이 클릭 1~2번 거리.
2. **작업 중심 흐름**: 파일 구조(디렉토리 트리)가 아니라 "하려는 일"(풀기·복습·등록) 중심으로 화면을 조직.
3. **학습 현황 상시 가시성**: 최근 점수·오답·찍음·추이가 홈에서 바로 보임.

## 2. 정보 구조(IA)와 네비게이션

### 채택안: 좌측 전역 사이드바 + 단계형 드릴다운(하이브리드)

```
[해시 라우트]
#/                     → 온보딩(닉네임 미설정 시) 또는 대시보드
#/cert/{분야}/{자격증}   → 자격증 상세 (기출 목록 + 업로드)
#/solve/{분야}/{자격증}/{시험ID} → 풀이 (몰입 모드; 제출 후 같은 화면에 결과 섹션)
```

```
네비게이션 흐름
온보딩 ──(닉네임 설정)──► 대시보드 ──(자격증 카드/사이드바)──► 자격증 상세 ──(기출 행)──► 풀이 ──(제출)──► 결과(풀이 화면 내)
                                 ▲            브레드크럼/사이드바로 복귀            │
                                 └────────────────────────────────────────────────┘
```

- **좌측 사이드바(전역)**: 분야로 그룹핑된 자격증 목록 + "＋ 자격증 등록". 현재 라우트 하이라이트. 풀이 화면에서는 자동 접힘(토글 버튼으로 재노출) — 몰입 모드 확보.
- **브레드크럼(콘텐츠 상단)**: `대시보드 / {분야} / {자격증} / {시험ID}`.
- **참고안(드릴다운 단독) 대비 변경 이유**: 사용자가 좌측 사이드바 도입을 명시 요청했고, 이 앱의 계층이 얕아(분야>자격증 2단) 사이드바가 화면 전환 없는 시험 간 전환을 제공한다. 대시보드 카드 그리드는 유지하되 사이드바가 이를 보완(중복 아님 — 카드는 현황 요약 포함, 사이드바는 빠른 점프).
- **결과를 별도 라우트로 만들지 않는 이유**: 결과는 제출 응답(qnet:attempt-submitted detail)에 의존하는 휘발 데이터. 새로고침 재진입 불가한 URL을 만들지 않는다. 과거 점수는 자격증 상세의 "내 기록"과 대시보드 현황 카드가 담당.

## 3. 화면별 와이어프레임

### 3.1 온보딩 (닉네임 미설정 시 전체 화면)

```
┌────────────────────────────────────────────┐
│                                            │
│              [Q]  Q-Net 기출 풀이           │
│   기출 PDF를 풀고, 자동 채점받고, 오답을      │
│   복습하는 로컬 스터디 앱입니다.              │
│                                            │
│   닉네임 선택  [ 기존 참여자 ▾ ]             │
│   또는 새 닉네임  [__________] [시작하기]     │
│                                            │
└────────────────────────────────────────────┘
```
- 기존 참여자 = `/api/repo` participants 합집합(드롭다운). 새 닉네임 입력과 병행.
- 성공 시 `#/` 대시보드 진입. 이후 닉네임 변경·삭제는 헤더 닉네임 메뉴에서(기능 라운드 2의 usage 조회·삭제 모달 로직 재사용, 기능 동일 유지).

### 3.2 대시보드 (`#/`)

```
┌ 헤더: [Q] Q-Net 기출 풀이            ☾ | junseong ▾ ┐
├ 사이드바 ─┬──────────────────────────────────────────┤
│ 정보처리   │ ▶ 이어풀기 — 2023-1-필기 (응답 12/80)     │  ← 드래프트 있을 때만(최상단 CTA)
│  · 정보처리기사│                                        │
│  · 정보처리산업기사│ 내 현황                             │
│ ＋ 자격증 등록│ ┌────────┬────────┬────────┬────────┐   │
│           │ │최근 점수 │ 오답 X  │ 찍음    │ 추이     │   │
│           │ │ 72.5 합격│  14개   │  6개   │ ╱╲╱ SVG │   │
│           │ └────────┴────────┴────────┴────────┘   │
│           │ 자격증                                    │
│           │ ┌──────────────┐ ┌──────────────┐        │
│           │ │ 정보처리기사    │ │ 정보처리산업기사│  ...   │
│           │ │ 기출 3 · 시도 5 │ │ 기출 0        │        │
│           │ │ 최근 72.5 합격  │ │ 시작 전       │        │
│           │ └──────────────┘ └──────────────┘        │
│           │ ┌ ＋ 새 자격증 등록(dashed 카드) ┐          │
└───────────┴──────────────────────────────────────────┘
```
- **이어풀기 CTA**: localStorage 드래프트 미러(`qnet-drafts`) 기반. 클릭 → 해당 풀이 라우트. 서버 `GET /api/draft/:examId`로 존재 검증 후 표시(불일치 시 미러 정리).
- **내 현황**: 자격증별 `GET /api/attempts?grade&cert` 합산(자격증 수가 적은 로컬 앱이라 병렬 fetch 허용). 최근 시도 점수+판정, 총점 추이 스파크라인(인라인 SVG). 기록 없으면 "아직 기록이 없습니다 — 첫 기출을 풀어보세요" 빈 상태.
  - **오답 X·찍음 타일의 계약 공백(확정 동작)**: 현행 `GET /api/attempts` 목록 계약은 요약 필드(시험·시도·총점·합격여부·풀이일)만 제공하고 문항 단위 집계(X수·찍음수)가 없다. 이번 라운드는 서버 수정 금지·신규 API 금지 제약이므로 두 타일은 **응답에 해당 필드가 있으면 합산, 없으면 '— / 상세 미제공' 정직 폴백**(0 위장 금지)이 확정 동작이다. 후속 서버 라운드에서 `listAttempts`가 attempt frontmatter의 X수·찍음수를 노출하면 프론트 무수정으로 숫자가 채워진다(전방호환 소비 로직 기구현).
- **자격증 카드**: `/api/repo` + `GET /api/exams` 병합(기출 수·채점 가능 수) + 최근 시도. 클릭 → 상세.
- **새 자격증 등록**: dashed 카드 클릭 시 인라인 폼(분야 datalist + 자격증명 → `POST /api/certs`, 기능 라운드 2 구현 재사용).

### 3.3 자격증 상세 (`#/cert/{분야}/{자격증}`)

```
┌ 브레드크럼: 대시보드 / 정보처리 / 정보처리기사        ┐
│ 정보처리기사                       [＋ 기출 업로드]   │
│ ┌ 기출 목록 ──────────────────────────────────┐     │
│ │ [2023] [1회] [필기]  채점 가능   내 최근 72.5  ▶ │     │
│ │ [2022] [3회] [필기]  열람만(정답 미등록)      ▶ │     │
│ │ [2024] [2회] [실기]  정답 등록 후 열람 가능    ▶ │     │
│ └─────────────────────────────────────────────┘     │
│ ▼ 기출 업로드(접이식): 드롭존(드래그오버 상태 표시)     │
│    업로드 → 판독 → 검증 → 등록 단계 진행 표시          │
│    실패 시: 정답 수동 입력 폼(기존 로직 재배치)         │
└─────────────────────────────────────────────────────┘
```
- 기출 행: 연도·회차·구분 **뱃지 3종** + 상태 뱃지(`채점 가능`=ok / `열람만`=warn / `잠김`=muted) + 내 최근 점수·시도 수(`/api/attempts`) 표시.
- 클릭(열람 가능 시) → 풀이 라우트. 잠김 행은 클릭 시 정답 등록 안내 + 수동 입력 폼으로 스크롤.
- 업로드는 접이식 패널로 강등(주 동선은 "풀기"). 드롭존 + 단계 진행 표시(업로드→판독→검증→등록), NDJSON/응답 기반 상태 갱신은 기존 로직 유지.

### 3.4 풀이 (`#/solve/...`) — 몰입 모드

```
┌ 솔브바(sticky): ← 정보처리기사 | 2023-1-필기   응답 12/80 ▓▓▓░░ 12:34  [제출·채점] ┐
├ (사이드바 자동 접힘) ──────────────┬ OMR 레일(고정 스크롤) ──────┤
│                                  │ 소프트웨어 설계 (1-20)        │
│        PDF 뷰어 (넓게)            │  1  ① ② ③ ④   [찍음] [개념·챗]│
│        페이지 카드 연속 스크롤       │  2  ① ② ③ ④   …            │
│                                  │  3 …                        │
│                                  │ (현재 문항 하이라이트)          │
└──────────────────────────────────┴─────────────────────────────┘
                          [도구 패널: 우측 슬라이드 오버(개념/챗), ESC 닫기]
```
- **솔브바**: 뒤로(브레드크럼 축약) + 시험ID + 진행률 바 + 경과시간 + 제출 버튼. 항상 보임.
- **키보드**: `1~4` 현재 문항 답, `g` 찍음 토글, `↑/↓` 문항 이동, `c` 개념, `/` 챗. input/textarea 포커스 시 전부 비활성. 현재 문항은 OMR 레일에서 액센트 하이라이트 + 자동 스크롤.
- **도구 패널(개념·챗)**: 우측 슬라이드 오버(뷰어 위로 겹침, 폭 400px, ESC/✕ 닫기). 문항 컨텍스트 칩(`#12 · 2023-1-필기`) 표시. 챗은 말풍선 + 스트리밍 커서(▍) + 승인 다이얼로그(목적지 체크박스+설명) 기존 계약 유지.
- 드래프트 자동저장(디바운스 800ms)·이어풀기·제출 계약 그대로. 저장 시 localStorage 미러 갱신, 제출 성공 시 미러 제거.

### 3.5 결과 (풀이 화면 내, 제출 후 스크롤 진입)

```
┌ 채점 히어로: 총점 72.5   [합격]   확신 정답률 81% ┐
│ 과목별: 소프트웨어설계  ▓▓▓▓▓▓▓░ 70  ┊40 과락선   │
│         소프트웨어개발  ▓▓▓░░░░░ 35  ┊(위험색)    │
│ 시도 추이: ╱╲╱ (인라인 SVG, 1차 60 → 2차 72.5)    │
│ ── 다음 단계 ──                                   │
│ ① 오답 키워드 입력(인라인 폼, 건너뛰기 가능)         │
│ ② [오답 개념 보기] ③ [챗으로 질문]                 │
│ [원본(답지 포함) 보기]                             │
└──────────────────────────────────────────────────┘
```
- `qnet:attempt-submitted` 수신 계약 유지. 과목 게이지에 40점 과락 기준선 마커, 과락 과목 위험색.
- "오답 개념 보기/챗으로 질문" = X·찍음 문항 목록에서 해당 문항 도구 패널을 여는 바로가기(복습 퍼널).

### 3.6 공통 요소

- **헤더**: 브랜드 + 우측: 테마 토글(☾/☀, localStorage `qnet-theme`, `:root[data-theme]`) + 닉네임 칩 드롭다운(변경/삭제 — 기능 라운드 2 모달 로직 이동, JS 스타일 주입은 CSS로 이관).
- **CLI 상태**: 상단 배너 유지하되 정상(ok)일 때는 5초 후 자동 축소(헤더의 작은 상태 도트로 대체) — 경고(미설치·감사 위반)는 유지 표시.
- **토스트**(`components/toast.js`, 신규): 저장 완료·등록 완료 등 성공/정보 알림. `qnet:toast` 커스텀 이벤트로 발행. fs-change 자동 갱신은 무음 유지.
- **빈 상태/로딩/에러**: 스켈레톤(카드·목록), 행동 유도 버튼 있는 빈 상태, 사람이 읽는 에러 문구.

- 유지 계약: 서버 API 스키마 그대로(신규 API 없음 — 대시보드 통계는 기존 API 병합·localStorage 미러로 해결), `qnet:attempt-submitted`/`qnet:fs-change` 의미 유지, `store.js apiFetch`의 X-QNet-Token 유지, 채점·드래프트 계산 로직 불변.
- 기능 라운드 2 산출물(닉네임 usage·삭제, `/api/certs` 등록 폼) 기능 동일 이식 — JS `<style>` 주입(`nickname-ui-styles`)은 제거하고 CSS 파일로 이관.

## 5. 디자인 시스템

- v1 토큰 체계 계승(서페이스 3·텍스트 3·상태 4종·radius·shadow·간격) + **다크 토글**: `@media(prefers-color-scheme)` 기본값 + `:root[data-theme=light|dark]` 명시 오버라이드(토글 3상태 auto→light→dark).
- 두 모드 모두 본문 대비 4.5:1 이상. 폰트 스택·줄간격 1.6 유지. 숫자 tabular-nums.
- 접근성: 포커스 링, 모달 포커스 트랩+ESC, 아이콘 버튼 aria-label, 클릭 타깃 ≥32px, 키보드만으로 대시보드→풀이→제출→결과 완주.

## 6. 검증 계획 (완료 조건)

- 전제: 기능 라운드 2(병렬 세션) 완료·정지 후 착수(파일 경합 방지). 착수 직전 최신 파일 재독 필수.

## 부록 — 검증 결과 기록 (구현 후 QA)

### 첫 화면 3초 테스트 자체 평가 — **통과**

대시보드 스크린샷(라이트, 드래프트 존재 상태) 기준:
- 한 줄 설명 없이도 용도 파악 가능: 좌측 사이드바(분야>자격증), "내 현황"(최근 점수·판정 뱃지·추이), "자격증" 카드 그리드(기출 수·채점 가능 수·최근 점수)로 "기출을 풀고 채점받는 학습 앱"임이 즉시 드러남.
- 다음 행동 1클릭 거리: 최상단 **이어풀기 CTA**(진행 중 시험ID·응답 진행률 표기) 또는 자격증 카드 클릭. 신규 사용자는 dashed "＋ 새 자격증 등록" 카드.
- 첫 방문(기록 없음) 상태도 빈 현황 카드에 "첫 기출을 풀어보세요 → [자격증 살펴보기]" 행동 유도 버튼 제공.
- 잔여 약점: 온보딩 직후 자격증·기출이 모두 0인 극초기 상태에서는 "기출 업로드까지의 여정"이 2단계(등록→상세 업로드) — 로컬 스터디 앱 특성상 허용.

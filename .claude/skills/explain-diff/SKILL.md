# explain-diff — 코드 변경을 이해하기 위한 설명 문서

> 원안: Geoffrey Litt, "Understanding is the new bottleneck" (2026) 및 공개 스킬 `explain-diff`.
> 이 저장소용으로 한국어화하고 웹 앱(`web/`) 맥락에 맞췄다.

## 왜 이 스킬이 있는가

에이전트가 코드를 대량으로 작성하는 시대에는 **이해가 병목**이 된다. 이해는 두 가지 이유로 필요하다.

- **검증(verification)**: 이게 올바른가? — 에이전트가 점점 잘하게 되는 영역.
- **참여(participation)**: 다음 아이디어를 내려면 지금 무슨 일이 벌어지는지 알아야 한다 — **인간이 계속 맡아야 하는 영역.**

이 스킬의 목표는 후자다. 원시 diff(raw diff)는 "무슨 일이 있었나"의 재료일 뿐이다. 좋은 설명은 **배경 → 직관 → 코드 → 이해 확인**의 순서로, 팀이 이 변경을 나에게 가르치려고 커리큘럼을 짠 것처럼 만든다.

**속도 조절 장치 규칙**: 문서 맨 아래 퀴즈를 통과하지 못하면, 그 변경을 남에게 리뷰 요청하거나 다음 작업으로 넘어가지 않는다. 퀴즈는 "빨리 가기"가 아니라 "제대로 이해하며 가기"를 위한 브레이크다.

## 대상 diff 수집

인자로 받은 대상에서 변경 내용을 모은다. 명시가 없으면 사용자에게 무엇을 설명할지 확인한다.

- 커밋/범위: `git show <해시>` / `git diff <base>..<head>` / `git diff HEAD~1`
- 스테이지된 변경: `git diff --cached`
- 작업 중 변경: `git diff`
- PR: `gh pr diff <번호>` (gh 사용 가능 시)

변경 파일 목록(`git diff --name-only ...`)을 먼저 보고, **주변 코드를 폭넓게 탐색**해 배경을 파악한다. diff만 보고 쓰지 말 것 — 이 변경이 건드리는 기존 시스템을 읽어야 배경 섹션을 쓸 수 있다.

## 문서 구조 (4개 섹션 고정)

1. **배경 (Background)** — 이 변경과 관련된 기존 시스템을 설명한다. 독자의 사전 지식을 모르므로 두 겹으로 쓴다: ① 입문자를 위한 깊은 배경(이미 아는 사람은 건너뛸 수 있다고 명시), ② 이 변경에 직결되는 좁은 배경.
2. **직관 (Intuition)** — 변경의 **본질**을 먼저 전달한다. 세부보다 직관이 우선이다. 장난감 데이터로 구체적 예시를 들고, 그림·다이어그램을 아낌없이 쓴다. ("이 커밋의 핵심은 X를 Y로 바꿔 Z가 되게 하는 것이다"를 한 문장으로 먼저 말한다.)
3. **코드 (Code)** — 변경을 이해 가능한 순서로 묶어 고수준 워크스루를 한다. 파일을 알파벳순으로 나열하지 말고, **읽기 좋은 순서**로 재배열해 각 덩어리 앞에서 "여기서 무슨 일이 일어나는가"를 말한다(literate diff).
4. **이해 확인 퀴즈 (Quiz)** — 이 변경을 실제로 이해했는지 확인하는 5문항. 중간 난이도(함정이 아니라, 내용을 이해해야 풀리는 수준). 객관식 + 클릭 시 정오 판정과 피드백.

## 퀴즈 작성 원칙 (Andy Matuschak, "How to write good prompts")

- **집중적**: 한 문항은 하나의 아이디어만 묻는다.
- **정밀함**: 무엇을 답해야 하는지 애매하지 않게.
- **다루기 쉬움**: 이해했다면 풀 수 있는 수준. 지엽적 함정 금지.
- **노력을 요함**: 질문 문장에서 답이 그대로 유추되면 안 된다.
- **정답 위치 균형**: 보기 순서를 무작위화하고, 5문항의 정답이 특정 번호에 몰리지 않게 분산한다(길이·문법·위치로 정답이 새지 않도록).
- 각 보기 클릭 시: 정답 여부 + **왜 맞고 왜 틀린지** 한 줄 해설을 보여준다.

## 출력 형식

- **단일 자체 완결 HTML 파일**(CSS·JS 인라인 포함, 외부 의존성 0). 목차가 있는 한 페이지. 최상위 구조에 탭을 쓰지 말고 한 줄로 스크롤되게. 휴대폰에서도 읽히게 기본 반응형.
- **라이트/다크 양쪽** 스타일. `@media (prefers-color-scheme: dark)`로 다크를 기본 신호로 삼는다.
- 저장 위치·파일명: **`.omc/explain/`** 아래(gitignore되는 재생성 가능 영역), 파일명은 **오늘 날짜 접두** `YYYY-MM-DD-<슬러그>.html`. 시간순 정렬되고 버전관리에서 빠진다. 디렉토리가 없으면 만든다.
  - 예: `.omc/explain/2026-07-21-concept-view-tag-index.html`
- **다이어그램은 ASCII 금지**, 항상 HTML로 그린다(간단한 div·표·목록). 유용한 계열:
  - 앱 UI를 아주 단순화한 그림(UI 변경 설명용).
  - 컴포넌트 간 데이터 흐름 시스템 다이어그램 — **반드시 예시 데이터를 넣는다.**
  - 소수의 다이어그램 계열을 정해 문서 전체에서 재사용한다.
- **코드 블록은 `<pre>` 태그**를 쓴다. 커스텀 div를 쓴다면 CSS에 반드시 `white-space: pre` 또는 `pre-wrap`을 넣는다(안 그러면 브라우저가 줄바꿈을 다 합쳐버린다). 저장 전 각 코드 블록의 CSS를 훑어 확인한다.
- 핵심 개념·정의·주의할 엣지 케이스는 **콜아웃 박스**로.
- 문체: 명료하고 흐름이 매끄러운 한국어. 섹션 간 전환이 자연스럽게. (Martin Kleppmann처럼 쉽고 우아하게.)

## 실행 후

1. 생성한 HTML의 절대경로를 알려주고, 브라우저에서 여는 법을 안내한다(`open <경로>` 또는 파일 더블클릭).
2. "퀴즈를 먼저 풀어보고, 막히는 문항이 있으면 그 부분을 다시 설명해 달라"고 안내한다.

## 규칙

- 저장소 소스 코드는 **읽기만** 한다. 이 스킬은 설명 문서만 만들고 코드를 고치지 않는다.
- 산출물은 `.omc/explain/`(gitignore) 밖에 커밋하지 않는다 — 개인용 일회성 학습 자료다.
- Notion 판은 이 저장소에서 쓰지 않는다(로컬 md/git이 진실 원천). HTML 판만 쓴다.
